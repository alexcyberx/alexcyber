Scenario

CorpX security team received an alert: a machine at 192.168.1.78 made repeated DNS requests to an unknown external domain during a 20-second window. A packet capture was taken at the network egress point.

Someone on the inside may have used DNS to smuggle data out. Your job is to open the capture, trace the suspicious traffic, and reconstruct what was sent.

Analysis Path
1
Open the file, check protocol summary
In Wireshark: Statistics then Protocol Hierarchy. Note the mix of TCP, UDP, and DNS traffic. Get a feel for what protocols are present.
2
Isolate DNS traffic
Apply display filter dns. Look at the Info column. Most queries are normal internal lookups. One machine queries an external domain repeatedly and gets NXDOMAIN back each time.
3
Identify the suspicious source and domain
Find which LAN IP is sending queries to 8.8.8.8 for an unknown external domain. Note the parent domain name.
4
Extract subdomain labels in timestamp order
Each query uses a different subdomain label. Write them down in the order they were sent (sort by time). There are 4 labels total.
5
Decode and concatenate to get the flag
Join the 4 labels in order and decode the result. The flag will be in the standard ACX{...} format.
Hints
Filter by dns and look at the source IP column. One machine on the 192.168.1.x subnet is making all 4 suspicious queries. It is not the same machine doing normal internal DNS lookups.
The parent domain is data.corpx-sync.net. The 4 queries each use a different subdomain prefix. Each prefix is a chunk of encoded data. Collect them in order of timestamp.
The labels are Base32 encoded (lowercase, no padding). Join the 4 labels into one string, convert to uppercase, pad with = signs until the length is a multiple of 8, then Base32 decode. Online tools: cyberchef.org with the "From Base32" recipe works well.
Submit Flag
Flag Captured
DNS exfiltration tunnels data through subdomain labels, bypassing firewalls that permit DNS but block HTTP. Each NXDOMAIN response here was expected since the domain was never meant to resolve. Defenders should alert on repeated NXDOMAIN responses to the same unknown parent domain from a single internal host.
