The briefing

What the security team already knows, and what they are asking you to confirm.

What happened

On the night of November 8, the CorpX web server saw a burst of requests to paths that do not exist on the normal site, followed by requests carrying SQL syntax against a real page. Someone mapped the server, then attacked what they found.

What you have

The full Apache combined-format access log for that day, 2,156 lines. It is unfiltered, real traffic mixed with the attack, so you will need to separate normal visitors and bots from the one IP that mattered.

What you need to find

Three things: the attacker's IP address, the tool that was used to break in (not the recon tool, the one that actually exploited the page), and the exact path it exploited.

Open it in a text editor, or use grep and awk if you want to work the way the security team does.

Where to start looking

Not the answer, just a sane way into 2,156 lines.

Isolate the failures

Legitimate visitors mostly get 200s. A source hammering 403s and 404s across dozens of paths in the same second is not a person browsing the site.

grep " 403 \| 404 " corpx_access.log

Read the user agent

Real browsers announce themselves as Chrome, Safari, Firefox. Scanning and exploitation tools usually announce themselves too, just not to a browser.

awk -F'"' '{print $6}' corpx_access.log | sort -u

Follow one IP forward in time

Once you find the address behind the noisy requests, pull every line from that IP in order. The story changes partway through, from mapping the server to attacking one specific page.

grep "45.33.32.156" corpx_access.log | less

Read carefully: the same IP uses two different tools at two different times. The first tool only looks for pages that exist. The second tool sends SQL syntax to one of the pages the first tool found. The flag wants the tool that actually got results, not the one that just went looking.
