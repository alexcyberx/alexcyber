WEB-02 Stored Cross-Site Scripting
Medium
Time 00:00
Scenario

You are performing a web application pentest on EchoBoard, an internal employee message board used by CorpX Ltd.

During recon you noticed the app fetches and renders comments without any output encoding. The server also leaks its render_mode in API responses.

Your goal: inject a stored XSS payload that persists in the database and executes in a victim browser on page load. Retrieve the flag from the server response.

Target Endpoints
Available
POST /api/lab/web02/comment GET /api/lab/web02/comments POST /api/lab/web02/reset
Request body (POST /comment)
Content-Type: application/json { "username": "...", "message": "<inject here>" }
Objectives
  • 1 POST a comment to the message board
  • 2 Inject a valid XSS payload in the message field
  • 3 Trigger payload execution on GET /comments
  • 4 Retrieve the flag from the server response
Hints
Both fields are stored, but only one is reflected in the rendered board. Intercept the POST in Burp Suite and observe which field content appears in the GET /comments response body. Check the server_info.render_mode field in the response — it tells you exactly how the frontend inserts comments.
The server leaks render_mode: "innerHTML" — any HTML that causes script execution will work. Think beyond just <script> tags. Event handlers on HTML elements (onerror, onload, onfocus) and SVG-based payloads also trigger XSS when inserted via innerHTML.
This is stored XSS, not reflected. The payload is stored in the database on POST. The flag is returned when the server detects a persisted XSS payload at retrieval time — simulating what happens when a victim loads the page and the stored payload executes in their browser.
